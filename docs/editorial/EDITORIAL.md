# EDITORIAL.md — 寫作方法論

> 移植自 taiwan-md 的編輯體系,針對技術文件在地化。
> 目標:幾十個人(和 AI)寫出來的文章,讀起來像同一個編輯室出品。

## 核心原則

1. **可跟做(Reproducible)**:每篇教學必須讓讀者「照著做就能成功」。
   - 指令一律放在標好語言的 code fence 裡,可直接複製。
   - 每個關鍵步驟後面寫「預期輸出」或「怎麼確認成功了」。
   - 可能失敗的地方,就地給檢查點與對策,或連到 troubleshoot 文章。
2. **有來源(Sourced)**:任何技術宣稱都要對得到官方文件、release notes 或 issue。
   frontmatter 的 `upstream_refs` 至少一筆;內文重要宣稱用 `[^n]` 腳註(見 CITATION-SYSTEM.md)。
3. **有版本(Versioned)**:frontmatter 必填 `hermes_version`(適用版本範圍)與
   `last_verified`(你最後一次實測的日期)。寧可寫窄,不要寫「全版本適用」然後過期害人。
4. **說人話**:先講「為什麼要做這件事、做完會得到什麼」,再講怎麼做。
   不要從第一行就開始貼指令。

## 反塑膠條款(AI 灌水偵測)

以下寫法會被 article-health 掃描器標記,PR 審稿時會被要求重寫:

- 空洞開場:「在當今快速發展的 AI 時代……」「眾所周知……」
- 萬用結尾:「總而言之」「希望這篇文章對你有幫助」
- 無資訊修飾:「扮演著重要的角色」「不可或缺的一環」「值得一提的是」
- **列表傾倒(list-dump)**:整篇都是條列沒有敘事。列表只該出現在
  「步驟」「選項比較」「檢查清單」三種場景。
- 沒有具體例子的抽象描述。技術文章的每個概念都應該有一段實際指令、
  設定檔片段或輸出結果來落地。

## 文章開場的三種模式(擇一)

1. **場景式**:從讀者正在遇到的狀況切入(「你剛裝好 WSL2,打了 `hermes` 卻出現 command not found——」)
2. **結果式**:先展示做完會得到什麼(截圖、輸出),再回頭教
3. **問題式**:從一個具體的錯誤訊息或 issue 出發

禁止:定義式開場(「Hermes Agent 是一個……」)——那是 concepts/ 分類專屬的寫法。

## 結尾

教學文的結尾只有兩種:「下一步做什麼」(連到下一篇)或「常見問題」。
不寫總結段。
